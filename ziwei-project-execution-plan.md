# ESP32-S3 紫微斗数排盘项目执行方案

目标：

- 在 `ESP32-S3-Touch-LCD-2.8B` 开发板上实现一个可触摸操作的紫微斗数排盘应用
- 支持输入年月日时与性别
- 生成本命盘
- 支持切换查看 `大限 / 流年 / 流月 / 流日`
- 为每个层级提供对应的分析页面

这份方案默认基于你已经下载的官方 IDF 例程继续演进，而不是从零裸写。

## 1. 项目目标拆解

我们要做的不是单一页面，而是一套小型本地应用，包含三类能力：

1. 输入能力
2. 排盘计算能力
3. 呈现与分析能力

对应成产品形态，就是：

- 一个输入页
- 一个命盘页
- 一组运限切换页
- 一组分析详情页

## 2. 推荐技术路线

硬件与软件栈建议：

- 开发框架：`ESP-IDF`
- UI：`LVGL`
- 显示驱动：沿用官方例程的 `esp_lcd + ST7701S`
- 触摸驱动：沿用官方例程的 `GT911`
- 时间选择：页面输入，不依赖 RTC
- 数据存储：先内存计算，后续可选 `NVS` 或 `TF 卡`

原因：

- 这块板子的 RGB 屏与触摸，官方例程已经打通
- LVGL 适合做多页面、可滑动、可切换的命盘界面
- 排盘逻辑是纯计算，很适合独立成 service 层

## 3. 建议项目范围

建议分两期做，不要一开始把所有命理细节全塞进去。

### 第一期：可用版本

实现：

- 输入出生年月日时、性别
- 生成本命盘
- 支持查看命宫、身宫、十二宫、十四主星、四化
- 支持切换大限、流年、流月、流日
- 每个层级提供基础分析页

不做或弱化：

- 复杂神煞全量支持
- 太多流派差异配置
- 花哨动画

### 第二期：增强版本

增强：

- 更多辅曜、神煞、流耀
- 多流派配置
- 结果保存
- 历史命盘管理
- 更完整分析模板

### 4.2 分层职责

`board` 层：

- 只做硬件初始化和板级接口
- 不写业务逻辑

`service` 层：

- 只做排盘、流年、格式化、分析

`ui` 层：

- 只负责界面创建、刷新和交互

`app` 层：

- 管理页面切换、状态同步、用户流程

## 5. 核心数据模型

建议先定义统一的数据结构，再写代码。

### 5.1 输入模型

```c
typedef struct {
    int year;
    int month;
    int day;
    int hour_index;
    int minute;
    int gender;   // 0 female, 1 male
} ziwei_birth_input_t;
```

### 5.2 本命盘模型

至少包含：

- 农历月日
- 年干支、月干支、日干支、时干支
- 命宫、身宫
- 十二宫
- 十四主星
- 常用辅曜
- 四化

### 5.3 运限模型

至少包含：

- 当前选中层级：本命 / 大限 / 流年 / 流月 / 流日
- 当前选中值
- 对应宫位映射
- 对应四化
- 对应分析摘要

### 5.4 页面状态模型

建议统一一个 `app_state`：

- 当前输入数据
- 当前本命盘结果
- 当前运限结果
- 当前页面
- 当前选中的宫位
- 当前选中的分析页签

## 6. 页面规划

建议 UI 分成 6 个页面。

### 6.1 首页

功能：

- 进入排盘
- 查看最近一次命盘
- 进入设置

### 6.2 输入页

功能：

- 输入公历年月日
- 选择时辰
- 选择性别
- 点击“开始排盘”

推荐交互：

- 年月日用滚轮或数字步进器
- 时辰用 12 宫式按钮列表或滚轮
- 性别用开关按钮

### 6.3 本命盘页

功能：

- 显示十二宫格
- 在宫位里显示主星/辅曜/四化
- 高亮命宫、身宫
- 点击宫位进入详情分析

推荐布局：

- 中间 12 宫环形或传统方阵布局
- 底部工具栏切换“本命 / 大限 / 流年 / 流月 / 流日”

### 6.4 运限选择页

功能：

- 选择某个大限区间
- 选择某一年
- 选择某一月
- 选择某一日

建议：

- 大限用横向列表
- 流年用年份滚轮
- 流月、流日用联动选择器

### 6.5 分析页

功能：

- 展示当前宫位的解释
- 展示当前运限叠加解释
- 显示关键词、简述、建议

建议结构：

- 顶部：当前层级和宫位
- 中部：星曜与四化摘要
- 底部：分析文本分页

### 6.6 设置页

预留：

- 立春/春节换年
- 子时口径
- 闰月口径
- 字体大小
- 背光亮度

## 7. 排盘服务拆分

建议按 4 个 service 模块拆开。

### 7.1 `ziwei_calendar`

负责：

- 公历转农历
- 干支换算
- 时辰映射
- 虚岁计算

### 7.2 `ziwei_chart`

负责：

- 命宫、身宫
- 十二宫
- 五行局
- 十四主星
- 常用辅曜
- 本命四化

### 7.3 `ziwei_horoscope`

负责：

- 大限
- 小限
- 流年
- 流月
- 流日

### 7.4 `ziwei_analysis`

负责：

- 宫位分析模板
- 星曜组合摘要
- 本命与运限叠加说明

第一期建议先做“规则模板式分析”，不要一开始上复杂自然语言生成。

## 8. 分析内容策略

你后面说的“对应分析页面”，建议先用规则库驱动，而不是完全自由生成。

### 8.1 第一版分析来源

可按三层组织：

1. 宫位基础含义
2. 星曜基础含义
3. 运限叠加规则

例如：

- 命宫 + 紫微：人格、主导性、气场
- 财帛宫 + 武曲：重财务管理、偏执行与积累
- 流年化忌落某宫：提示压力点

### 8.2 输出形式

建议每页拆成固定块：

- 关键词
- 简要判断
- 关注点
- 建议

这样更适合小屏阅读。

## 9. 从当前官方例程迁移的方式

建议不要直接在现有官方 demo 上硬改，而是：

1. 新建一个独立工程目录
2. 把官方例程里的底层驱动复制为 `board` 层
3. 删除演示用 `LVGL_Example.c`
4. 新建自己的 `ui/pages`
5. 新建自己的 `service`

### 9.1 可以直接复用

- LCD 初始化
- GT911 触摸接入
- LVGL 初始化
- 背光控制
- TF 卡挂载
- RTC 驱动

### 9.2 只参考，不建议直接继承

- `main.c` 里所有逻辑堆在一起的写法
- `LVGL_Example.c`
- 全局变量直接给 UI 读的模式
- 无线扫描展示逻辑

## 10. 执行阶段建议

推荐分 6 个阶段。

目标：

目标：

## 12. 主要风险

### 12.1 命理规则风险

风险点：

- 不同流派在年界、子时、闰月、四化和神煞上有差异

处理方式：

- 第一版固定一套规则
- 后续再做配置项

### 12.2 UI 密度风险

风险点：

- 紫微斗数一屏信息量极大
- 2.8 寸屏幕不适合一次塞太多字

处理方式：

- 命盘页只显示核心信息
- 详情放到分析页

### 12.3 性能风险

风险点：

- 大量文本渲染、中文字体、频繁重绘可能带来卡顿

处理方式：

- 减少动画
- 只局部刷新
- 字体精简

### 12.4 代码耦合风险

风险点：

- 如果直接在官方 demo 上堆代码，后面会很难维护

处理方式：

- 一开始就分层

## 13. 第一版功能清单

建议第一版明确只做下面这些：

- 输入出生年月日时、性别
- 本命盘展示
- 大限展示
- 流年展示
- 流月展示
- 流日展示

1. 先新建工程，不直接污染官方 Demo
2. 先把板级驱动层搬过来
3. 先实现“输入 -> 本命盘 -> 分析”最短闭环
4. 再加大限、流年、流月、流日
5. 最后再做分析内容丰富化
